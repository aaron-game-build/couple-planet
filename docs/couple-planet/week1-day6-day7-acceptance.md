## 2. Day6 联调脚本（按顺序执行）

前置准备：

- 设备 A 登录账号 A
- 设备 B 登录账号 B
- 两端均连接测试环境

## Step 1：注册/登录验证

- A 注册并登录
- B 注册并登录
- 双方调用 `GET /auth/me`

通过条件：

- 均返回 200
- 返回 user id 与本账号一致

## Step 2：绑定验证

- A 调用 `POST /relations/invite-code` 生成邀请码
- B 使用邀请码调用 `POST /relations/bind`
- A/B 均调用 `GET /relations/current`

通过条件：

- relationId 一致
- partner 信息正确

## Step 3：即时消息主链路

- A -> B 发送 10 条消息
- B -> A 发送 10 条消息
- 双方拉取历史并验证排序

通过条件：

- 20 条消息全部可见
- 无丢失、无重复、顺序正确（按 created_at）

## Step 4：已读回写

- B 打开 A 的 10 条消息并调用 read
- A 拉取历史确认 readAt 更新

通过条件：

- 已读数量与 messageIds 对齐
- readAt 字段被写入

## Step 5：弱网与重试

- A 断网后尝试发送 3 条消息
- 恢复网络后重发失败消息

通过条件：

- UI 有 `failed` 状态
- 重发成功后不重复落库（clientMsgId 幂等）

## Step 6：异常与鉴权

- 使用无效 token 调接口
- 使用已绑定账号再次绑定

通过条件：

- 返回预期错误码（非 500）

## 3. Day7 验收与复盘脚本

## 3.1 回归清单

- [ ] 注册流程
- [ ] 登录流程
- [ ] 绑定流程
- [ ] 消息发送/拉取
- [ ] 已读回写
- [ ] 失败重发

## 3.2 指标采样（至少 1 天）

- `绑定完成率 = 绑定成功用户数 / 进入绑定页用户数`
- `消息发送成功率 = sent 消息数 / 发送请求总数`

## 3.3 缺陷分级

- P0：阻断主链路（无法注册/绑定/发消息）
- P1：功能可用但体验严重受损
- P2：可延后修复的小问题

## 4. Go/No-Go 判定

## Go（进入 Week2）必须同时满足

- [ ] 两个账号可稳定完成注册、绑定、互发消息
- [ ] 消息发送成功率 >= 99%
- [ ] P0 缺陷 = 0

## No-Go（不进入 Week2）任一命中

- [ ] 存在 P0 缺陷
- [ ] 消息发送成功率 < 99%
- [ ] 绑定流程存在高概率失败

No-Go 处理动作（只做这三件）：

1. 修复绑定冲突与错误码不一致
2. 修复消息重复与丢失
3. 修复弱网失败重试体验
