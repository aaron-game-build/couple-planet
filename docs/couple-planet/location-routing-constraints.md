# 马车信封定位与路径模拟技术约束（MVP）

## 1. 目标与非目标

目标：

- 在不持续后台追踪用户位置的前提下，提供“可信的在途体验”。
- 控制电量、权限申请和地图调用成本。
- 保证送达状态在多端一致。

非目标（MVP 不做）：

- 秒级实时轨迹追踪
- 真正随用户移动实时重算全路径
- 高精度交通路况驱动 ETA 动态波动

## 2. 定位策略（离散采样）

位置采样时机：

1. 发送前采样（发送方当前位置）
2. 接收方最近可用位置读取（服务端缓存）
3. 旅程中低频修正（默认每 30 分钟最多 1 次）
4. 到达前校准（剩余里程 <= 2km 或剩余时间 <= 3 分钟）

约束：

- 不申请“持续后台高精度定位”作为 MVP 必选权限。
- 用户未授权定位时，允许手动选城市级位置 + 固定旅程模板。
- 每用户每日定位读取上限（服务端可配置）：120 次。

## 3. 路径与 ETA 计算

## 3.1 初始路径生成

- 输入：起点、终点、时间戳、交通模式（默认驾车）
- 外部依赖：地图路径规划 API
- 输出：总里程、预计时长、路径摘要（Polyline 可选）

ETA 基础公式（MVP）：

- `baseEta = routeDuration * moodFactor * eventFactor`
- `moodFactor` 默认 1.0，可由节日活动调整（0.9~1.1）
- `eventFactor` 由随机事件驱动（0.8~1.2）

边界约束：

- 最短 ETA：3 分钟
- 最长 ETA：24 小时

## 3.2 旅程中修正

修正规则：

- 仅在低频修正窗口触发（默认 30 分钟）或关键节点（网络重连、应用重启）。
- 修正幅度阈值：若新 ETA 与旧 ETA 偏差 < 10%，仅更新内部，不打扰用户。
- 偏差 >= 10% 时，更新旅程页 ETA，并记录一次“路线调整事件”。

## 4. 状态同步模型

状态机：

- `Created` -> `Dispatched` -> `InTransit` -> `Arrived` -> `Opened`
- 异常分支：`Failed` / `Expired`

一致性策略：

- 服务端为信封状态唯一事实源（Source of Truth）。
- 客户端只做乐观展示，不可单方推进终态。
- 所有状态变更写入事件日志表（append-only）。

幂等要求：

- `arriveEnvelope(envelopeId)` 必须幂等。
- 重复回调或重复推送不应造成重复奖励。

## 5. 失败与降级策略

常见失败场景与兜底：

- 地图 API 超时：使用城市级直线距离估算 ETA（保底算法）。
- 路径规划失败：回退固定速度模型（50km/h）+ 地理距离估算。
- 推送失败：客户端下次启动通过拉取接口补齐状态。
- 网络抖动：客户端本地显示“信号波动，马车仍在路上”，避免硬错误文案。

过期策略：

- `Created` 后 5 分钟未成功派发则转 `Failed`，可重试。
- `InTransit` 超过最长 ETA + 30 分钟转 `Expired`，系统自动补偿并说明原因。

## 6. 成本与性能约束

地图调用预算（建议）：

- 每封信封最多 3 次路径规划调用：
  - 创建时 1 次
  - 旅程中修正最多 1 次
  - 到达前校准最多 1 次

缓存策略：

- 同城高频 OD（origin-destination）路径结果缓存 24 小时。
- 近似起终点（误差半径 500 米）可复用缓存路由摘要。

性能指标（服务端）：

- ETA 计算接口 p95 < 300ms（命中缓存）
- 路径规划接口失败率 < 1%
- 信封状态写入成功率 >= 99.99%

## 7. 隐私与合规约束

- 只保存业务所需最小定位信息（坐标、采样时间、精度级别）。
- 默认不展示精确家庭地址，不输出可反向推断的原始轨迹。
- 用户可在设置中关闭定位并切换“手动城市模式”。
- 解绑后按数据策略清理或封存定位记录（见关系解绑规范）。

## 8. 可验证测试清单

功能测试：

- 无定位权限情况下可完成信封发送与送达。
- 弱网环境下状态可恢复并最终一致。
- 多端登录同账号时，旅程状态一致。

压力测试：

- 1 万并发在途信封下，状态推进无积压（目标 < 2 秒）。
- 路径服务 10% 错误注入时，降级策略仍保障 > 98% 可送达。

验收门槛：

- 信封送达成功率 >= 99%
- 平均单封地图调用次数 <= 2
- 日均定位采样次数不超过预算阈值
